## 4. Mandatory 7-Point Pitch Deck Structure

Every submission's Presentation Deck / Application **must** clearly and compulsorily contain these 7 points (per Focus Areas guidance — this is stated as a "critical part" of the jury's evaluation process):

1. **Problem Statement** — specific and concrete, not vague. ("India's supply chain is slow" = bad. "Last-mile medical delivery in rural areas is delayed by 48 hours" = good.)
2. **Technology Differentiation** — why is the tech different/better than existing solutions? What's the core enabler?
3. **Impact Metrics** — real numbers, not hand-waving (e.g. emissions reduced, energy saved, waste reduced, cost saved — or, for this project, alerts flagged / analyst-hours saved / satellites coverable).
4. **Theme Alignment** — explicit explanation of fit to the stated hackathon theme(s).
5. **Collaborations** — mention any NGO, company, government body, startup, or academic institution partnership or data usage; adds credibility. (Only claim real collaborations — do not fabricate.)
6. **Scalability** — how does the current small-scale prototype grow to a full India-level implementation? Full roadmap required.
7. **Elevator Pitch** — a powerful 1–2 minute-readable summary of innovation, uniqueness, and impact.

See `PRD.md §16` for this project's specific content mapped to each of these 7 points.

## 5. Judging Philosophy: Perceive → Reason → Execute

The jury's core evaluation mantra is these three words. A strong submission's software should visibly address all three:

| Pillar | Meaning | How this project maps |
|---|---|---|
| **Perceive** | System understands unstructured, complex real-world data (images, text, sensors) | Ingesting live TLE orbital data from Celestrak |
| **Reason** | System makes good decisions in edge cases / unpredictable situations — not just simple if/else | Propagating orbits (SGP4), screening for close approaches, computing an explainable two-factor risk score |
| **Execute** | System takes autonomous action with minimal human intervention | Generating an alert + a recommended monitoring/action window without manual analysis |

**Design rule flowing from this:** make this mapping *visually explicit* in the app itself (`PerceiveReasonExecuteStrip`, per `Design.md §6`), not just something the presenter explains verbally.

## 8. Our Own Engineering Discipline Rules (self-imposed, not from the organizer — carried over and strengthened from the team's original honesty-first instinct)

These aren't organizer rules, but the team should treat them as equally non-negotiable, because they are the difference between a credible sovereignty pitch and an oversold one:

1. **Never claim sub-kilometer precision.** TLE/SGP4 propagation has well-documented, growing error over time-since-epoch. State this plainly in the deck and in the app's `DisclaimerBar`.
2. **Always label the distance threshold a "monitoring threshold," never a "collision threshold."** These are different things and conflating them is the single easiest way to lose credibility with a technically literate judge.
3. **Never fabricate impact numbers.** Illustrative/roadmap-stage estimates (per `PRD.md §12.2`) must be presented as illustrative, not as measured results.
4. **Never claim a collaboration/partnership that doesn't exist**, even though the deck explicitly rewards mentioning collaborations — an unearned claim is a bigger risk than a blank slide.
5. **Say the non-goals out loud in the deck.** Explicitly stating what this tool is *not* (full-catalog screening, operational-grade SSA, precision maneuver planning) is a stated credibility strategy, not a weakness — see `PRD.md §5.4`.
6. **Keep the scoring formula visible and inspectable at all times in the UI.** No black-box risk scoring, ever, in this version of the product.
7. **Re-fetch fresh TLE data immediately before every demo recording or live presentation.** Non-negotiable operational habit, not a one-time task.
8. **Keep backend (`snake_case`) and frontend (`camelCase`) schemas manually in sync** per `Schema.md §5`, since this codebase has no shared codegen step — a silent mismatch here is a common last-minute demo-breaking bug.

## 9. Quick Do's and Don'ts

| Do | Don't |
|---|---|
| Verify NORAD IDs for Indian satellites against the live Celestrak catalog at build time | Hardcode satellite IDs from memory/assumption |
| Widen the threshold / switch constellation if zero alerts are found | Panic-debug the screening code before trying the parameter fix |
| Show the risk-scoring formula with real numbers plugged in | Hide the formula behind a "trust us" black box |
| Submit well before the Aug 21, 2026 Round 1 deadline | Wait until the last hour — there is zero room to fix mistakes post-submission |
| Keep API keys (if any are ever added, e.g. for Space-Track) in a `.env`, excluded via `.gitignore` | Commit any credential to the repository |
| Frame illustrative impact numbers as illustrative | Present roadmap estimates as measured results |
