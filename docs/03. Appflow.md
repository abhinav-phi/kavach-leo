# Application Flow
## Project KAVACH — Indigenous LEO Satellite Conjunction-Assessment Tool

**Companion to:** `PRD.md`, `TechSpec.md`, `Design.md`
**Version:** 2.0

---

## 1. High-Level Flow

```
Landing / App Load
      │
      ▼
Fetch cached satellite list (GET /api/satellites) ──► show TLE freshness badge
      │
      ▼
User adjusts params (threshold / window / step) in Control Panel  [optional — sane defaults preload]
      │
      ▼
User clicks "Run Screening"  ──►  POST /api/screen
      │
      ▼
Loading state ("Perceive" stage — fetching + propagating)
      │
      ├── Success ──► Results state: 3D scene + Alerts panel populated
      │                     │
      │                     ▼
      │              User clicks an alert card
      │                     │
      │                     ▼
      │              Camera flies to that conjunction; Score Breakdown panel opens ("Reason" stage)
      │                     │
      │                     ▼
      │              Monitoring window shown ("Execute" stage)
      │
      └── Failure (TLE fetch or propagation error) ──► Error banner + fallback to cached data
```

## 2. Screen-by-Screen Detail

### 2.1 Landing / Header
- Title: "KAVACH-LEO" + one-line tagline ("Sovereign LEO Collision Risk Monitor").
- TLE freshness badge (e.g. "Data fetched 12 min ago" or "⚠ Using cached data — 3h old").
- Perceive → Reason → Execute strip, all three stages shown greyed-out/inactive until a run starts.

### 2.2 Control Panel (always visible, left or top strip)
- Satellite group selector (defaults to the pre-built mixed working set — dense constellation + named Indian satellites; advanced users can change the constellation).
- Threshold slider (10–100 km, default 50 km, labeled "Monitoring Threshold — not a collision threshold").
- Window/step inputs (defaults 24h / 5min, collapsed under "Advanced" so the default view isn't cluttered).
- **Run Screening** button — primary call to action.

### 2.3 Loading State
- Triggered immediately on "Run Screening" click.
- Perceive stage lights up in the status strip ("Ingesting TLE data...", then "Propagating orbits...").
- 3D scene shows the Earth and satellite markers appearing progressively (not a blank spinner) — keeps the screen visually alive even mid-computation, important for a live judge-facing demo (dead air reads badly).
- Target: this state should rarely last more than a few seconds per the performance targets in `TechSpec.md`; if it exceeds ~5s, show a lightweight progress indicator so it doesn't look frozen.

### 2.4 Results State (the core screen — see `Design.md` for exact visual treatment)
Two-pane layout:
- **Left/main (≈65–70% width): OrbitScene** — rotating 3D Earth, all selected satellite orbit paths drawn, flagged pairs highlighted (glow + pulse), idle auto-rotate when nothing is selected.
- **Right/side (≈30–35% width): AlertsPanel** — sorted list of `AlertCard`s (HIGH risk first), each showing pair names, min distance, `RiskBadge`, and a "View" affordance.
- `DisclaimerBar` pinned at the bottom, always visible, never scrolled out of view.
- Reason stage lights up in the status strip the moment alerts are computed and scored.

### 2.5 Alert Selected (the "winning moment")
- Clicking an `AlertCard` (or clicking the highlighted conjunction directly in the 3D scene — both should work) triggers:
  1. Camera smoothly flies/zooms to frame both satellites at their time of closest approach (~1–2 second eased animation — see `Design.md §5` for timing).
  2. A connecting line + pulsing marker appears at the point of closest approach.
  3. `ScoreBreakdown` panel slides in: shows the actual proximity-factor and closing-speed-factor numbers plugged into the formula (FR-4.3 — no black box).
  4. Monitoring window is displayed with start/end timestamps.
  5. Execute stage lights up in the status strip.
- This is the single moment the whole build is optimized around — it should require zero narration for a judge to understand "these two are close, this is how close, this is when to watch."

### 2.6 Deselect / Return to Overview
- Clicking elsewhere in the scene or a "Back to overview" affordance returns the camera to the idle auto-rotate framing, `ScoreBreakdown` panel closes, Reason/Execute strip states dim back to their "computed but not currently focused" style (not fully reset — the run already happened).

### 2.7 Error / Fallback Flow
- If `POST /api/screen` fails (network, TLE fetch failure, propagation error on a bad TLE):
  - A visible, honest warning banner appears: "Live TLE fetch failed — showing cached data from [timestamp]." (Never fail silently — this is a stated reliability requirement in `PRD.md`.)
  - If cached data exists, the pipeline still runs against it and the Results state loads normally, just with the warning persistent.
  - If no cached data exists at all (first run, no network), show a clear empty-state message with a retry button — do not show a blank or broken screen.

### 2.8 Threshold/Param Change Mid-Session
- Adjusting the threshold slider after a run has completed re-triggers screening automatically (debounced) against the already-propagated position data where possible (avoids a full re-fetch/re-propagate for a threshold-only change — an optimization worth implementing since judges often ask "what if you loosen/tighten this" live).

## 3. Persona Journeys

### 3.1 Priya (Smallsat Ops Lead)
Opens the app → sees her constellation's satellites already in the default working set → runs screening with default 50km threshold → sees one HIGH alert involving two of her satellites → clicks it → reads the monitoring window → knows exactly when to watch.

### 3.2 Dr. Rao (Cubesat Program Lead, educational use)
Opens the app → widens the window to 48h to show students a broader picture → opens `ScoreBreakdown` on a MEDIUM alert to walk through the formula live as a teaching moment → this is why the formula is shown, not hidden.

### 3.3 Regulatory Analyst
Opens `/api/screen`'s underlying data via the Swagger docs (Round 2 stretch) → cross-checks the tool's flagged pairs against a foreign-sourced conjunction data message → values that nothing here is a black box.

### 3.4 The Judge
Sees the screen for the first time, unguided → within ~10 seconds identifies "this shows satellites, one pair is highlighted red, there's a risk score" without narration → the presenter then narrates the Perceive/Reason/Execute strip and the sovereignty framing on top of what's already visually obvious. This ordering (visual-first, narration-second) is intentional — see `PRD.md §2` on why the demo is the differentiator.
