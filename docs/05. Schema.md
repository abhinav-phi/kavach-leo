# Data Schema
## Project ARGUS-LEO — Indigenous LEO Satellite Conjunction-Assessment Tool

**Companion to:** `TechSpec.md`, `AppFlow.md`
**Version:** 2.0

All schemas below are the source of truth for both the FastAPI `pydantic` models (backend) and the TypeScript interfaces (frontend) — keep the two in sync manually since this is a small hackathon codebase without a shared codegen step; note this explicitly in `Rules.md`'s engineering discipline section.

---

## 1. Core Entities

### 1.1 `Satellite`
```json
{
  "name": "string — display name from TLE line 0",
  "norad_id": "string — NORAD catalog number",
  "tle_line1": "string — raw TLE line 1",
  "tle_line2": "string — raw TLE line 2",
  "epoch": "ISO 8601 datetime — TLE epoch (data generation time)",
  "hours_since_epoch": "float — computed at request time, drives freshness badge",
  "is_indian_asset": "boolean — true for the curated named Indian satellites (Cartosat/RISAT/EOS/IRNSS etc.)"
}
```

**Validation rules:**
- `norad_id` must be numeric-string, 1–6 digits.
- `tle_line1`/`tle_line2` must each be exactly 69 characters (standard TLE line length) — reject and log, don't crash, if a fetched line fails this check (handles Celestrak formatting edge cases gracefully, per FR-2.3 in `PRD.md`).
- `epoch` is derived from the TLE, not user-supplied.

### 1.2 `PositionSeries`
```json
{
  "satellite_name": "string",
  "timestamps": ["ISO 8601 datetime", "..."],
  "eci_km": [[x, y, z], "... one triple per timestamp, Earth-Centered Inertial frame, kilometers"]
}
```
- `timestamps` and `eci_km` are always the same length.
- Position magnitude sanity range: 6,700–8,000 km for LEO objects (see `TechSpec.md §8` sanity check) — any value outside this range indicates upstream parsing/timescale error and should fail loudly in tests, not silently in production.

### 1.3 `ConjunctionAlert`
```json
{
  "id": "string — short hash of sat_a + sat_b + screening run timestamp",
  "sat_a": "string — satellite name",
  "sat_b": "string — satellite name",
  "min_distance_km": "float, 2 decimal places",
  "time_of_closest_approach": "ISO 8601 datetime — from the fine-refinement pass, see TechSpec.md §4.2",
  "relative_speed_kms": "float — finite-difference estimate at closest approach",
  "risk_score": "float, 0.0–100.0, 1 decimal place",
  "risk_level": "enum: LOW | MEDIUM | HIGH",
  "score_breakdown": {
    "proximity_factor": "float 0.0–1.0",
    "proximity_weight": 0.7,
    "closing_speed_factor": "float 0.0–1.0",
    "closing_speed_weight": 0.3
  },
  "monitoring_window": {
    "start": "ISO 8601 datetime",
    "end": "ISO 8601 datetime"
  }
}
```
- `score_breakdown` is always included in full (never truncated/omitted) — this field exists specifically to back the "no black-box scoring" claim in `PRD.md` FR-4.3 and the `ScoreBreakdown` UI component in `Design.md §6`.

### 1.4 `ScreeningRequest`
```json
{
  "window_hours": "int, default 24, range 12–48",
  "step_minutes": "int, default 5, range 1–15",
  "threshold_km": "float, default 50.0, range 10.0–100.0",
  "satellite_group": "string, default 'starlink'"
}
```

### 1.5 `ScreeningResponse`
```json
{
  "generated_at": "ISO 8601 datetime",
  "stale": "boolean — true if served from cache after a live-fetch failure",
  "params": "ScreeningRequest (echoed back)",
  "positions": "map of satellite_name -> PositionSeries",
  "alerts": "array of ConjunctionAlert, sorted by risk_score descending"
}
```

## 2. File / Cache Schema

### `data/tle_cache.txt`
Plain text, 3 lines per satellite (name, TLE line 1, TLE line 2), matching Celestrak's native TLE format — chosen deliberately for human-readability and easy diffing/debugging over a binary or JSON cache.

### `data/tle_cache_meta.json`
```json
{
  "fetched_at": "ISO 8601 datetime",
  "group": "string",
  "count": "int",
  "schema_version": "1"
}
```

---

## 3. Schema Versioning

- `schema_version` field included in cache metadata and API responses (default `"1"` for this build). Not expected to change during the hackathon window, but included from day one so a Round 2 change doesn't silently break a Round 1 recorded demo/artifact if both are referenced later.

---

## 4. Internal Python Data Shapes (for implementers)

- `positions: dict[str, np.ndarray]` — key is satellite name, value is a `(3, N)` NumPy array of ECI km positions (matches the original v1.0 draft's shape choice — kept because it's the natural output of `skyfield`'s `.position.km` and vectorizes cleanly for `screen.py`'s pairwise distance calculation).
- `alerts: list[dict]` — built in `screen.py`, enriched by `risk_score.py`, serialized to the `ConjunctionAlert` JSON shape above at the API boundary in `api.py`.

## 5. Frontend TypeScript Mirror (for implementers)

```ts
interface Satellite {
  name: string;
  noradId: string;
  epoch: string;
  hoursSinceEpoch: number;
  isIndianAsset: boolean;
}

interface ScoreBreakdown {
  proximityFactor: number;
  proximityWeight: number;
  closingSpeedFactor: number;
  closingSpeedWeight: number;
}

interface ConjunctionAlert {
  id: string;
  satA: string;
  satB: string;
  minDistanceKm: number;
  timeOfClosestApproach: string;
  relativeSpeedKms: number;
  riskScore: number;
  riskLevel: "LOW" | "MEDIUM" | "HIGH";
  scoreBreakdown: ScoreBreakdown;
  monitoringWindow: { start: string; end: string };
}
```

Keep field names in `camelCase` on the frontend and `snake_case` on the backend/wire format (FastAPI/pydantic default) — translate at the API-call boundary in one place (a small `mapAlert()` function), not scattered across components.
