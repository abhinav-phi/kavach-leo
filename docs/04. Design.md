# Design Specification
## Project KAVACH — Indigenous LEO Satellite Conjunction-Assessment Tool

**Companion to:** `PRD.md`, `TechSpec.md`, `AppFlow.md`
**Version:** 2.0

---

## 1. Design Principles

1. **"Mission control," not "SaaS dashboard."** This is a sovereignty pitch — it should feel like it belongs in an operations room, not like a generic analytics product. Think ISRO Mission Control Centre displays, not a startup landing page.
2. **The visual is the argument.** Per `PRD.md §2`, the 3D scene carries more persuasive weight than any slide of text. Every design decision should ask "does this make the converging-orbits moment land harder, or does it dilute it?"
3. **Legible at a distance, in a hurry.** Judges see this on a shared screen or projector for a few minutes. No small text, no relying on subtle color differences alone, no cluttered chrome competing with the globe.
4. **Honesty is a design value, not just a copy value.** The disclaimer/caveat section is designed to be genuinely visible (not a grey footnote), because the project's whole credibility argument rests on "we tell you exactly what this is and isn't."

## 2. Visual Identity

### 2.1 Color Palette

| Role | Color | Notes |
|---|---|---|
| Background (scene + app shell) | Deep space navy/near-black — `#05070D` to `#0B0F1A` | Sets the "mission control" tone; makes glowing elements pop |
| Earth sphere | Muted blue-teal gradient, low-poly or subtly textured — `#1C4E66` base | Stylized, not photorealistic — avoids needing external imagery/tokens (see `TechSpec.md §2.2`) |
| Normal orbit lines | Soft cyan/white — `#8FD9FF` at low opacity (~40%) | Recedes visually so flagged orbits stand out |
| Flagged/alert orbit + marker | Alert red — `#FF3B3B`, with a soft outer glow | The one color that should dominate attention when present |
| Medium risk accent | Amber — `#FFB020` | Used in `RiskBadge`, not in the 3D scene (keep the scene's red reserved for the highest-signal moment) |
| Low risk accent | Green/cyan — `#3DDC97` | `RiskBadge` only |
| Indian-asset marker accent | A restrained saffron/white/green tricolor-inspired ring around the marker dot — used sparingly, only as a small accent ring, never as a loud banner | Ties the sovereignty narrative visually to specific satellites without looking gimmicky |
| UI chrome (panels, text) | Near-white `#E8ECF1` on the dark background; secondary text `#8B93A7` | High contrast for distance-legibility |

### 2.2 Typography
- **Primary UI font:** a clean geometric sans (e.g. "Inter" or "IBM Plex Sans") for panel text, labels, buttons.
- **Data/monospace font:** "IBM Plex Mono" or "JetBrains Mono" for numeric readouts (distance, scores, timestamps) — monospace numbers look "instrumented," reinforcing the mission-control feel, and align cleanly in tables.
- **Minimum body size:** 14px equivalent in the deployed app; risk scores and satellite names in the alert list no smaller than 16px — this is a distance-legibility requirement, not a preference.

### 2.3 Iconography
- Minimal, line-style icons only (satellite, alert triangle, clock for monitoring window). Avoid busy or skeuomorphic icon sets — they compete with the glow effects in the 3D scene.

## 3. Layout

```
┌─────────────────────────────────────────────────────────────────────┐
│  KAVACH-LEO   [TLE freshness badge]   [Perceive→Reason→Execute]     │  header
├─────────────────────────────────────────────┬───────────────────────┤
│                                             │ Alerts Panel          │
│                                             │ ┌───────────────────┐ │
│               3D Orbit Scene                │ │ AlertCard HIGH    │ │
│               (≈65-70% width)               │ ├───────────────────┤ │
│                                             │ │ AlertCard MED     │ │
│                                             │ ├───────────────────┤ │
│                                             │ │ AlertCard LOW     │ │
│  [Control Panel — collapsible strip]        │ └───────────────────┘ │
├─────────────────────────────────────────────┴───────────────────────┤
│  Disclaimer / caveat bar — always visible, never scrolled away      │
└─────────────────────────────────────────────────────────────────────┘
```

- The 3D scene and the alerts panel are **always visible together** (per FR-7.2 in `PRD.md`) — never hidden behind a tab. A judge should never have to click to "find" the visualization.
- On narrower/projector displays, the layout can stack (scene on top, panel below) but the scene retains visual priority (larger area, first in reading order).

## 4. 3D Scene Design (`OrbitScene`)

- **Earth:** a sphere (radius scaled consistently with the km-based orbit data) with a subtle latitude/longitude grid overlay at very low opacity — communicates "this is a real coordinate space," not decoration.
- **Orbit paths:** thin lines (see palette above) traced from each satellite's propagated position array. Line thickness ~1.5px normal, ~4px + glow for a flagged pair.
- **Satellite markers:** small glowing points at the satellite's *current* position along its path (or at the scrubbed time if a time-scrubber is added as a stretch feature); hover reveals a label (name + `is_indian_asset` tricolor ring if applicable).
- **Idle state:** slow, continuous auto-rotate around the Earth when nothing is selected — keeps the screen visually alive during any dead air in a live demo.
- **Alert highlight:** at the moment of closest approach for a flagged pair, render a short connecting line between the two satellites with a pulsing marker at its midpoint (2–3 second pulse loop, not distractingly fast).
- **Camera fly-to:** on alert selection, animate the camera position + look-at target over ~1.5s with an eased curve (not linear — a linear camera move reads as robotic) to frame both satellites and the connecting line comfortably in view, then hold.

---

## 5. Motion & Animation Guidelines

| Interaction | Duration | Easing | Purpose |
|---|---|---|---|
| Camera fly-to on alert select | ~1.5s | ease-in-out | The core "winning moment" — deliberate, not instant, so it reads as a discovery rather than a jump-cut |
| Alert marker pulse | 2–3s loop | ease-in-out (breathing, not blinking) | Draws the eye without being obnoxious on a long demo loop |
| Panel slide-in (`ScoreBreakdown`) | ~300ms | ease-out | Standard UI responsiveness, secondary to the camera motion |
| Idle globe auto-rotate | continuous, slow (~1 full rotation per 60–90s) | linear | Ambient, not a focal animation |

**Rule of thumb:** the *only* animation that should ever feel emphasized is the alert fly-to. Everything else is quiet by comparison, so the one moment that matters doesn't compete with itself.

## 6. Component Specs

### `AlertCard`
- Satellite pair names (monospace).
- Min distance in km (large, prominent number).
- `RiskBadge` (color-coded pill, LOW/MEDIUM/HIGH — text label always included, never color alone, for accessibility).
- Time-to-closest-approach, relative to "now" if within the window (e.g. "in 6h 45m").
- Click target is the whole card, not just a small button — easy to hit live during a demo.

### `RiskBadge`
- Pill shape, colored per §2.1, with the risk level as visible text (not just color) — satisfies color-blind accessibility and remains legible on a low-quality video recording.

### `ScoreBreakdown`
- Shows the formula from `TechSpec.md §4.4` with the actual computed numbers substituted in, e.g.:
  `Proximity: 0.82 × 70% = 57.4  +  Closing speed: 0.34 × 30% = 10.2  →  Score: 67.6 (MEDIUM)`
- This concrete, plugged-in presentation is what makes "no black-box scoring" a demonstrated fact rather than a claim.

### `ControlPanel`
- Threshold slider with the current value always shown numerically next to the handle, plus the persistent label "Monitoring Threshold (not a collision threshold)" — the caveat lives right next to the control that sets it, not buried elsewhere.
- Advanced params (window/step) collapsed by default to keep the primary view uncluttered.

### `DisclaimerBar`
- Fixed at the bottom, moderate-contrast (not screaming red, but genuinely readable — near-white text on the dark background, not tiny grey-on-black). Content: SGP4/TLE accuracy limitations + "screening/triage layer, not operational-grade" positioning, matching `PRD.md §4`'s non-goals language.

### `PerceiveReasonExecuteStrip`
- Three segments, each dims/lights based on current pipeline stage (see `AppFlow.md §2`). Small icon + label per stage. This is a deliberate, literal visualization of the judging rubric — not subtle, intentionally so.

## 7. Accessibility & Demo-Readability

- Never encode risk level by color alone — always pair with text label (LOW/MEDIUM/HIGH) and, where practical, an icon (e.g. dot count or triangle fill level).
- Maintain WCAG-reasonable contrast between text and the dark background (near-white text `#E8ECF1` on `#05070D`/`#0B0F1A` comfortably clears this).
- Test the full flow at actual projector/screen-share resolution before the final recording — text and controls that look fine on a dev laptop can become illegible when screen-shared at a lower effective resolution.
- Ensure the "which satellites are at risk" answer is visible within the first 10 seconds of the Results state loading, with zero interaction required — this is the NFR from `PRD.md §9` translated directly into a design constraint: the highest-risk `AlertCard` should be visually first (top of list, boldest badge) with no scrolling needed.

---

## 8. What NOT to Do (explicit anti-patterns, given the source material's warning)

- Do not ship a plain data table as the primary view with the 3D scene as an afterthought — the mentor guidance in the source material is explicit that this specific failure mode is what loses.
- Do not add so much motion/glow that the flagged conjunction stops being the single most eye-catching thing on screen.
- Do not hide the disclaimer in a tooltip or a collapsed "learn more" — visibility of the caveat is a stated credibility strategy, not boilerplate.
- Do not use CesiumJS's default photorealistic Earth imagery reflexively "because it looks cool" — it pulls in an external token dependency (see `TechSpec.md §2.2`) that isn't needed for a stylized, deliberately mission-control aesthetic that we control fully.
