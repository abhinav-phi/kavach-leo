## 1. Architecture Overview

Two independent services, talking over a small JSON REST API. This is a deliberate change from the original single-process Streamlit design — reasoning in §2.

```
┌─────────────────────────────┐        ┌───────────────────────────────────────┐
│   BACKEND (Python)          │        │   FRONTEND (React, Vite)              │
│                             │        │                                       │
│  fetch_tle.py  ───────────► │  HTTP  │  ControlPanel ──► calls /api/screen   │
│  propagate.py               │◄──────►│  OrbitScene (react-three-fiber)       │
│  screen.py                  │  JSON  │  AlertsPanel                          │
│  risk_score.py              │        │  DisclaimerBar                        │
│  api.py (FastAPI routes)    │        │  Perceive/Reason/Execute status strip │
└─────────────────────────────┘        └───────────────────────────────────────┘
          │
          ▼
   data/tle_cache.txt
   data/tle_cache_meta.json
```

**Data flow, end to end:**
`Celestrak TLE API → local cache → skyfield/SGP4 propagation → pairwise distance screening → two-factor risk scoring → JSON response → React state → react-three-fiber scene + alerts panel`

## 2. Key Technical Decision: Visualization Stack

### 2.1 The decision
Frontend visualization uses **React + react-three-fiber (Three.js) with `@react-three/drei` helpers**, not Streamlit + Plotly, and not raw CesiumJS.

### 2.2 Why this beats the two alternatives considered

| Option | Verdict | Reasoning |
|---|---|---|
| **Streamlit + Plotly** (original v1.0 choice) | ❌ Rejected | Fast to build, but produces a generic "rotatable scatter plot" look. The source mentor material is explicit: a plain table/chart dashboard will bore judges regardless of backend sophistication. Camera fly-to-alert animations, custom glow/pulse effects, and a "mission control" aesthetic are impractical in Plotly. |
| **CesiumJS** (mentor's suggestion) | ⚠️ Considered, not chosen as primary | Industry standard for globe/space rendering and genuinely excellent for this use case. Downsides for a hackathon: (1) best features (terrain, imagery) require a free Cesium ion access token — an external dependency and signup step that risks breaking the "runs anywhere, no keys" portability requirement (NFR in `PRD.md`) right before a demo; (2) steeper API surface for a team on a tight clock; (3) more weight than needed since we don't need photorealistic Earth imagery — a clean stylized globe sells the sovereignty/"mission control" aesthetic better anyway (see `Design.md`). |
| **react-three-fiber / Three.js** (chosen) | ✅ Chosen | No external token or account needed — fully self-contained, satisfies the portability NFR. Full creative control over the exact look in `Design.md` (dark space background, glowing orbit lines, pulsing red alert markers, camera fly-to). React-native component model fits a team already comfortable with JS/React tooling. Slightly more setup work than Plotly, but the payoff (the actual "winning moment" described in the source material) is worth it. |

**If the team is critically short on frontend time near the Round 1 deadline:** fall back to a documented **Plan B** — keep the FastAPI backend exactly as designed, and stand up a fast Streamlit + `pydeck` (deck.gl) dashboard instead of full React. `pydeck` supports a rotatable 3D view with custom layers and is a meaningfully better middle ground than flat Plotly, while staying inside a pure-Python, same-day build. This is a explicit, named fallback — not a silent scope cut — and should be flagged in the deck if used ("full 3D web scene is the Round 2 target; this is our Round 1 interim view").

### 2.3 Decision trigger
Attempt react-three-fiber first (start it as a Week-1 spike per `ImplementationPlan.md`). If, by the midpoint checkpoint in `Tracker.md`, the scene isn't rendering orbits + camera fly-to, switch to Plan B rather than risk an unfinished 3D scene on demo day.

## 4. Backend Module Design

### 4.1 `fetch_tle.py`
```python
import requests
from datetime import datetime, timezone
import json

CELESTRAK_URL = "https://celestrak.org/NORAD/elements/gp.php"

def fetch_group(group: str) -> list[list[str]]:
    """Fetch a named Celestrak group (e.g. 'starlink') as [name, line1, line2] triples."""
    resp = requests.get(CELESTRAK_URL, params={"GROUP": group, "FORMAT": "tle"}, timeout=15)
    resp.raise_for_status()
    lines = resp.text.strip().split("\n")
    return [lines[i:i+3] for i in range(0, len(lines), 3)]

def fetch_by_norad_id(norad_id: str) -> list[str]:
    """Fetch a single named satellite by NORAD catalog ID (used for real Indian satellites)."""
    resp = requests.get(CELESTRAK_URL, params={"CATNR": norad_id, "FORMAT": "tle"}, timeout=15)
    resp.raise_for_status()
    return resp.text.strip().split("\n")[:3]

def build_working_set(constellation_group="starlink", constellation_cap=15,
                       indian_norad_ids: list[str] | None = None) -> list[list[str]]:
    """
    Mixed dataset: a dense same-constellation subset (guarantees a genuine close
    approach) plus a handful of named Indian satellites (narrative relevance —
    these are flagged distinctly in the UI even if they don't personally
    trigger an alert). See Schema.md for the 'is_indian_asset' field this feeds.
    """
    sats = fetch_group(constellation_group)[:constellation_cap]
    for nid in (indian_norad_ids or []):
        try:
            sats.append(fetch_by_norad_id(nid))
        except requests.RequestException:
            continue  # degrade gracefully — one missing satellite shouldn't break the run
    return sats

def cache_to_disk(sats, path="data/tle_cache.txt", meta_path="data/tle_cache_meta.json"):
    with open(path, "w") as f:
        for name, l1, l2 in sats:
            f.write(f"{name}\n{l1}\n{l2}\n")
    with open(meta_path, "w") as f:
        json.dump({
            "fetched_at": datetime.now(timezone.utc).isoformat(),
            "count": len(sats),
        }, f)
```

**NORAD IDs for named Indian satellites** (Cartosat, RISAT, EOS, IRNSS series) must be confirmed against Celestrak's current catalog at build time — satellite designations and availability change; do not hardcode from memory without verifying at implementation time.

### 4.2 `propagate.py`
Same core approach as the original draft (`skyfield.api.EarthSatellite` + `ts.utc(...)`), with two additions:
1. **Freshness check**: compute `hours_since_epoch = (now - sat.epoch.utc_datetime())` per satellite; surface this to the API response so the UI can show a "data freshness" badge (FR-1.4).
2. **Fine-refinement pass** (FR-2.4): after the coarse 5-minute-step pass identifies a flagged pair's approximate closest-approach index, re-propagate just that pair over a ±15-minute window at a 10-second step and take the true minimum. Cheap (only runs for already-flagged pairs) and materially improves the credibility of the reported distance.

```python
def refine_closest_approach(sat_a, sat_b, ts, coarse_time, window_minutes=15, step_seconds=10):
    n_steps = int((window_minutes * 2 * 60) / step_seconds)
    offsets = [(-window_minutes*60) + i*step_seconds for i in range(n_steps)]
    times = ts.tt_jd(coarse_time.tt + [o/86400 for o in offsets])
    pos_a = sat_a.at(times).position.km
    pos_b = sat_b.at(times).position.km
    dist = ((pos_a - pos_b) ** 2).sum(axis=0) ** 0.5
    idx = int(dist.argmin())
    return times[idx], float(dist[idx])
```

### 4.3 `screen.py`
Same pairwise-combinations approach as v1.0 (`itertools.combinations` + `numpy.linalg.norm`), unchanged in spirit — it was already correctly scoped ("brute-force on a curated subset, full-catalog needs spatial pre-filtering — stated as roadmap"). Add: for every flagged pair, call `refine_closest_approach` before scoring.

### 4.4 `risk_score.py` — two-factor scoring (upgraded from v1.0's distance-only formula)

```python
def compute_risk_score(min_distance_km: float, relative_speed_kms: float,
                        threshold_km: float = 50.0,
                        ref_max_speed_kms: float = 15.0) -> tuple[float, str]:
    """
    Deterministic, explainable, two-factor score. Both factors and weights
    are surfaced in the UI — no black box.

    Proximity factor (70%): how far below threshold the min distance is.
    Closing-speed factor (30%): relative velocity at closest approach,
      normalized against a reference LEO relative-speed range (typical
      LEO-LEO relative speeds run roughly 0–15 km/s depending on orbital
      planes; same-plane objects trend low, crossing-plane objects trend
      high). Higher relative speed at a given distance means less
      reaction/monitoring time — a real, explainable urgency signal, not
      a fake-precision collision-probability model.
    """
    proximity = max(0.0, (threshold_km - min_distance_km) / threshold_km)
    closing_speed = max(0.0, min(1.0, relative_speed_kms / ref_max_speed_kms))

    score = round((proximity * 0.7 + closing_speed * 0.3) * 100, 1)
    level = "HIGH" if score >= 70 else "MEDIUM" if score >= 35 else "LOW"
    return score, level

def monitoring_window(time_of_closest_approach, risk_level: str) -> tuple[str, str]:
    buffer_minutes = {"LOW": 20, "MEDIUM": 30, "HIGH": 45}[risk_level]
    start = time_of_closest_approach - buffer_minutes_as_timedelta(buffer_minutes)
    end = time_of_closest_approach + buffer_minutes_as_timedelta(buffer_minutes)
    return start.isoformat(), end.isoformat()
```

Relative speed is computed as the derivative of position across the two refined-window samples closest to the identified minimum (finite difference), which is already available from the fine-refinement pass in §4.2 at negligible extra cost.

## 5. API Contract

Base URL (local dev): `http://localhost:8000`

### `GET /api/health`
Liveness check. Returns `{ "status": "ok", "tle_cache_age_minutes": 12 }`.

### `GET /api/satellites`
Returns the current working set with freshness metadata.
```json
{
  "generated_at": "2026-07-14T10:00:00Z",
  "satellites": [
    { "name": "STARLINK-30001", "norad_id": "58234", "epoch": "2026-07-13T22:10:00Z",
      "hours_since_epoch": 11.8, "is_indian_asset": false },
    { "name": "CARTOSAT-3", "norad_id": "44804", "epoch": "2026-07-12T04:00:00Z",
      "hours_since_epoch": 29.9, "is_indian_asset": true }
  ]
}
```

### `POST /api/screen`
Runs the full pipeline (propagate → screen → score) for the current working set.

Request body (see `Schema.md` for full JSON Schema):
```json
{
  "window_hours": 24,
  "step_minutes": 5,
  "threshold_km": 50.0
}
```

Response:
```json
{
  "generated_at": "2026-07-14T10:00:12Z",
  "params": { "window_hours": 24, "step_minutes": 5, "threshold_km": 50.0 },
  "positions": {
    "STARLINK-30001": { "timestamps": ["2026-07-14T10:00:00Z", "..."], "eci_km": [[x,y,z], "..."] }
  },
  "alerts": [
    {
      "id": "a1b2c3",
      "sat_a": "STARLINK-30001",
      "sat_b": "STARLINK-30045",
      "min_distance_km": 4.82,
      "time_of_closest_approach": "2026-07-14T16:45:10Z",
      "relative_speed_kms": 0.42,
      "risk_score": 88.3,
      "risk_level": "HIGH",
      "monitoring_window": { "start": "2026-07-14T16:00:10Z", "end": "2026-07-14T17:30:10Z" }
    }
  ]
}
```

### `GET /api/alerts/{alert_id}`
Returns a single alert with the full scoring breakdown (proximity factor, closing-speed factor, weights applied) for the detail panel — this is what makes FR-4.3 ("no black-box scoring") concrete in the UI.

## 7. Caching & Offline Resilience

- `data/tle_cache.txt` + `data/tle_cache_meta.json` are always written on a successful fetch.
- On fetch failure, backend serves the cached file and sets a `stale: true` flag in the API response; frontend renders a visible (not silent) warning banner — this satisfies the "Reliability" NFR in `PRD.md` and avoids ever showing fabricated-looking fresh data that isn't fresh.
- Before any recorded demo or live presentation, run a manual re-fetch (`POST /api/refresh-tle` — simple wrapper around `fetch_tle.build_working_set` + `cache_to_disk`) so the shown data is current.

## 8. Testing & Validation Strategy

- **Unit tests** (`pytest`) for `screen.py` and `risk_score.py` using synthetic position arrays with a known, hand-computed minimum distance — verifies the pairwise-distance and scoring math independent of real TLE data or network access.
- **Sanity check** on propagation output: assert all computed ECI position magnitudes fall in the 6,700–8,000 km range for LEO objects (Earth radius ~6,371 km + LEO altitude). Anything outside this range means the TLE parsing or timescale is wrong — this check should run automatically, not just be a manual eyeball step.
- **Manual validation**: run the full pipeline against a dense Starlink subset and confirm at least one genuine sub-threshold conjunction appears before relying on it for a demo recording.
- This testing section is what the mandatory Round-1/2 "Testing & Validation proof" deliverable (per `Rules.md`) should point to.

## 9. Security

- No API keys required for the core pipeline (Celestrak is open, no-auth) — nothing to leak.
- If Space-Track.org integration is ever added on the roadmap, credentials go in a `.env` file, excluded via `.gitignore`, never committed — called out explicitly because the competition rules (`Rules.md`) treat committed credentials as a hard violation.
- No user-submitted data is collected or stored; the tool only reads public orbital data.
